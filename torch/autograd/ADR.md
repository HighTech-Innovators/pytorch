# Architecture Decision Record: torch.autograd

## Architectural Role

The `torch.autograd` subsystem is responsible for **automatic differentiation** — the core mechanism that enables gradient-based optimization in PyTorch. It implements a **dynamic computation graph** that is constructed at runtime during forward execution, and a **reverse-mode autodiff engine** that computes gradients by traversing this graph backward.

This subsystem bridges user-facing training loops with low-level gradient computation, providing both imperative control (`.backward()`) and functional composition (`Function`, `apply()`).

## Responsibilities

### What This Subsystem Owns

1. **Computation Graph Construction** (`graph.py`, `variable.py`)
   - Recording operations as they execute
   - Tracking tensor dependencies
   - Maintaining the DAG of computation
   - Distinguishing leaf tensors from intermediate values

2. **Backward Pass Execution** (`__init__.py`, `graph.py`)
   - Traversing the computation graph in reverse topological order
   - Invoking backward functions for each operation
   - Accumulating gradients at leaf tensors
   - Managing gradient lifecycle and cleanup

3. **Gradient Tracking Control** (`grad_mode.py`)
   - Context managers (`torch.no_grad()`, `torch.enable_grad()`, `torch.inference_mode()`)
   - Global grad-enable/disable state
   - Forward-mode autodiff state management

4. **Custom Autograd Functions** (`function.py`)
   - `Function` class for user-defined operations with custom backward logic
   - `FunctionCtx` for saving intermediate values and accessing metadata
   - `apply()` entry point for executing custom functions

5. **Gradient Checking and Validation** (`gradcheck.py`)
   - Numerical gradient verification
   - Correctness testing infrastructure for backward implementations

### What This Subsystem Does NOT Own

- **Operator Implementation**: ATen operators define how forward computation is performed; autograd only records what was computed, not how
- **Memory Allocation**: Tensor storage and allocation are owned by ATen; autograd manages only the graph metadata
- **Numerical Computation**: The actual CPU/GPU kernels for forward and backward are in ATen/aten; autograd orchestrates their invocation
- **Distributed Training**: Though gradients flow through distributed training, the distributed backend (torch.distributed) manages collective synchronization

## Dependencies

### Upstream Dependencies (What Uses This)

- **User Training Code**: Scripts and models using `.backward()`, `loss.backward()`, optimizer steps
- **torch.optim**: Optimizers read `.grad` attributes to update parameters
- **torch.nn.Module**: Forward hooks and loss computation typically invoke `.backward()`
- **Custom Loss Functions**: User code often calls `.backward()` on scalar losses

### Downstream Dependencies (What This Uses)

- **ATen** (`aten/`): Each recorded operation has a backward function implemented in ATen or auto-generated by torchgen. When backward executes, it calls these ATen-defined backward functions.
- **torch._C** (C++ bindings): Core `Variable` class, graph node classes, backward engine are implemented in C++; Python layer is thin wrapper
- **torch.utils** (`profiler.py` uses profiling utilities): Limited interaction with profiling infrastructure

### Dependency Direction

```
User Code (train loop)
    ↓
torch.optim.Optimizer.step()
    ↓
torch.autograd.backward()
    ↓
ATen backward functions
```

The dependency is **pull-based**: autograd pulls backward implementations from ATen in reverse topological order, not push-based.

## Trade-offs and Design Decisions

### Dynamic Graph vs Static Graph

**Decision**: PyTorch uses dynamic computation graphs, constructed at runtime during forward pass.

**Trade-off**:
- ✅ **Advantage**: Enables Python control flow within forward pass (conditional branches, loops depend on tensor values)
- ✅ **Advantage**: Graph structure matches actual code flow
- ❌ **Disadvantage**: Graph not known until forward pass executes; more overhead per iteration
- ❌ **Disadvantage**: Harder to optimize globally; no advance knowledge of structure

**Evidence**: Chapter 02 demonstrates that `if (x.sum() > 0).item()` can affect graph structure at runtime. Implementation in `src/torch/autograd/variable.py` and `src/torch/autograd/graph.py` shows the ImperativeEngine recording executed operations.

### Reverse-Mode (Backprop) vs Forward-Mode Autodiff

**Decision**: Default mode is reverse-mode (backprop); forward-mode is opt-in via `torch.autograd.forward_ad`.

**Trade-off**:
- ✅ **Advantage**: Reverse-mode efficient for scalar loss (many inputs, one output) — typical in ML
- ❌ **Disadvantage**: Forward-mode more efficient for few inputs, many outputs — less common in training
- ✅ **Advantage**: Simpler mental model for users writing training loops

**Evidence**: `src/torch/autograd/forward_ad.py` provides forward-mode as separate module; default backward path uses reverse-mode implemented in C++ ImperativeEngine.

### Gradient Accumulation at Leaves

**Decision**: Gradients accumulate in `.grad` attribute of leaf tensors; intermediate tensors do not store gradients.

**Trade-off**:
- ✅ **Advantage**: Leaf tensors clearly marked as trainable parameters
- ✅ **Advantage**: Clear ownership boundary: only leaves accumulate
- ❌ **Disadvantage**: Cannot directly access gradients of intermediate tensors without `.retain_grad()`
- ❌ **Disadvantage**: Additional memory overhead if intermediate gradients needed

**Evidence**: `src/torch/autograd/__init__.py` line ~255 validates that `.grad` is only accessible on leaf tensors. Chapter 02 shows that intermediate tensors have `.grad_fn` but not `.grad`.

### Manual .backward() vs Automatic Graph Cleanup

**Decision**: User calls `.backward()` explicitly; graph is retained until next forward pass or `.detach()` is called.

**Trade-off**:
- ✅ **Advantage**: User has explicit control over when backprop happens
- ✅ **Advantage**: Can inspect intermediate values in forward pass before computing gradients
- ❌ **Disadvantage**: User must remember to call `.backward()`; easy to forget in complex code
- ❌ **Disadvantage**: Graphs retained in memory until cleanup; potential memory leak if backward not called

**Evidence**: `src/torch/autograd/__init__.py` defines the `backward()` function; there is no automatic trigger.

## Extension Boundaries

### Implementing Custom Autograd Functions

**Boundary**: The `Function` class defines the protocol for custom operations.

A user implements:
1. **`forward()` method**: Define how the operation is computed
2. **`backward()` method**: Define how gradients flow backward
3. **`save_for_backward()` call**: Specify which intermediate values are needed in backward

```python
class MyFunc(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x, y):
        ctx.save_for_backward(x, y)
        return x * y
    
    @staticmethod
    def backward(ctx, grad_output):
        x, y = ctx.saved_tensors
        return grad_output * y, grad_output * x

# Use it:
result = MyFunc.apply(input_x, input_y)
```

**Evidence**: `src/torch/autograd/function.py` defines `Function` base class; `apply()` method (line ~95) is the entry point.

### Adding Custom Hooks

**Boundary**: Register hooks on tensors to inspect or modify gradients during backward.

```python
def hook_fn(grad):
    print(f"Gradient: {grad}")
    return grad

tensor.register_hook(hook_fn)
loss.backward()  # Hook called during backward
```

**Evidence**: `src/torch/autograd/__init__.py` and `src/torch/_C` define `register_hook()` API.

### Extending Gradient Tracking Modes

**Boundary**: Context managers control whether gradients are tracked.

```python
with torch.no_grad():
    result = model(x)  # No grad_fn recorded
```

**Evidence**: `src/torch/autograd/grad_mode.py` implements context managers.

## Runtime Implications

### Lifecycle and Initialization

1. **Eager Initialization**: Graph begins recording immediately when first tensor with `requires_grad=True` is created
2. **Lazy Graph Construction**: Nodes (operations) added only when they execute
3. **Backward Trigger**: User calls `.backward()` explicitly
4. **Cleanup**: Graph and intermediate gradients discarded after backward completes (unless `.retain_grad()` called)

### Concurrency Behavior

**Thread Safety**:
- **Unsafe**: The autograd engine is not thread-safe; each thread must have its own backward pass
- **Evidence**: Chapter 02 notes that backward execution is single-threaded within a thread context

**Distributed Training**: 
- **Limited Support**: Distributed training synchronizes gradients via `torch.distributed.all_reduce()` after backward completes
- **Evidence**: `torch.distributed` integration happens at optimizer level, not within autograd

### Failure Behavior

1. **NaN/Inf Gradients**: `anomaly_mode.py` can detect operations that produce NaN gradients
2. **Gradient Underflow**: No special handling; gradients may become zero or very small
3. **Graph Memory Exhaustion**: Large graphs can exhaust memory; no automatic eviction

**Evidence**: `src/torch/autograd/anomaly_mode.py` provides `detect_anomaly()` context manager.

## Performance Implications

### Known Hotspots

1. **Graph Traversal in Backward**: Walking the DAG in reverse topological order is O(num_operations); large models create large graphs
2. **Gradient Accumulation at Leaves**: Atomically accumulating into `.grad` has memory barrier overhead
3. **Context Switching**: Each operation's forward wrapper adds overhead to check `requires_grad`

### Allocation Patterns

- **No Allocation in Core Loop**: Autograd itself allocates graph nodes at Python/C++ boundary during forward pass; during backward, mostly traversal with no allocation
- **Memory Peak**: Intermediate tensors remain in memory until backward completes or explicitly freed

### Synchronization Costs

- **No Internal Synchronization**: Backward pass is single-threaded
- **Distributed Synchronization**: Only when distributed backend is used; happens post-backward

## Ownership Boundaries

### State Owned by Autograd

1. **Computation Graph**: Entire DAG structure, edge relationships, operation metadata
2. **Leaf vs Intermediate Distinction**: Flag on each tensor recording whether it's a leaf
3. **Gradient Accumulator**: `.grad` attribute storage at leaf tensors
4. **Backward Function Registry**: Map of operations to their backward implementations

### State Borrowed from Other Subsystems

1. **Tensor Values**: Owned by ATen; autograd only references them
2. **Operator Implementations**: Forward and backward kernels owned by ATen
3. **Allocator**: Memory allocation strategy owned by ATen's allocator

### State Owned by Users

1. **Graph Structure**: Users determine which operations execute via Python control flow
2. **Requires_grad Flags**: Users set `requires_grad` on input tensors
3. **Backward Triggers**: Users call `.backward()` explicitly
